# 設定ページ「選択中: cerulean」問題の修正計画（ステップバイステップ）

## 現象の整理

- 設定を開いた直後、「選択中: **cerulean**」（[`BOOTSWATCH_SWATCH`](components/theme_palette.py) の先頭キー）や、保存テーマと違う見た目になる。
- **ユーザー確認（重要）**: ログインユーザーの DB（`theme_settings`）の実値は **cerulean でも minty でもない別テーマ**である。よって「DB が cerulean だから表示も cerulean」では説明できず、**設定ページのラベル／プレビュー Store が DB 正本を反映していない**ことが確定する。
- 既存対応（[`app.py`](app.py) の `_sync_nav_store_and_theme` で `bootswatch-theme` と [`theme-store`](app.py) を `load_theme()` に同期）は **グローバル CSS と localStorage** には効くが、**設定ページ専用 UI** は別経路のまま。

## Step 1: データ経路の切り分け（原因の確認）

次の3系統が独立していることをコード上で確認する。

| 経路 | 役割 | `/settings` 初回で更新されるか |
|------|------|--------------------------------|
| A | [`_sync_nav_store_and_theme`](app.py): `bootswatch-theme` + `theme-store` | はい（`_pages_location`） |
| B | [`pages/settings/index.py`](pages/settings/index.py): `theme-preview-name` の **静的 children**、`theme-preview-store` の **初期 data** | レイアウト生成時のみ |
| C | [`mark_active_card`](components/theme_utils.py): `theme-preview-store` を Input に active クラス付与 | Store 変更時 |

**結論**: B が **一度だけ** `load_theme()` を読む。A は B を更新しないため、B が誤っていれば「選択中: cerulean」のまま残る。

## Step 2: 誤表示になり得る条件（最有力）

[`load_theme()`](components/theme_utils.py) は `_infer_members_id()` が `None` のとき **Supabase を読まず** [`_load_theme_from_file()`](components/theme_utils.py)（`theme.txt`）へ落ちる。

- **DB が「cerulean / minty 以外」なのに UI が cerulean になる**場合、レイアウト時の `load_theme()` は **DB に到達していない**（`g.user_id` 未設定など）可能性が高い。その結果 `_load_theme_from_file()` が返す **`theme.txt` の内容**（例: cerulean）が、`theme-preview-name` / `theme-preview-store` の初期値にそのまま載る。
- 一方、pathname コールバック実行時には **`flask.g.user_id` が入った状態**で同じ `load_theme()` が走り、**DB の実テーマ**が `bootswatch-theme` / `theme-store` に使われ **A だけ正しい**、という **二重経路のズレ**が起き得る（環境・リクエスト順・Pages のレイアウト取得タイミングに依存）。

`DEFAULT_THEME` は [`minty`](services/theme_service.py) なので、**ファイルも無ければ cerulean にはならない**。表示が cerulean なら、レイアウト時点では **「`theme.txt` 由来の cerulean」**か **「別経路で文字列が上書きされている」**のいずれかを疑う（DB 実値が cerulean でない前提では、前者が典型）。

## Step 3: 修正方針（正本の一本化）

**方針**: `/settings` に遷移したあと（`_pages_location` の pathname が `/settings`）、**pathname 同期と同じ `load_theme()`** で次を上書きする。

1. [`theme-preview-store`](pages/settings/index.py) の `data` … **テーマ名の文字列**（DB 実値と一致する任意の Bootswatch 名）のみを格納（現状と同型）。
2. [`theme-preview-name`](pages/settings/index.py) の `children` … `f"選択中: {theme}"`。

これにより:

- [`mark_active_card`](components/theme_utils.py) が `theme-preview-store` の更新で再実行され、**正しいカードに `active`** が付く。
- ラベルが静的 HTML の誤値のまま残らない。

実装場所の候補（いずれか一つでよい）:

- **[`components/theme_utils.py`](components/theme_utils.py) の `register_theme_callbacks` 内**に新規コールバックを追加（テーマ関連の集約として自然）。
- または [`app.py`](app.py) に追加（pathname 同期の隣だが、Outputs が増えすぎるため `theme_utils` 推奨）。

**コールバック仕様（案）**:

- `Input("_pages_location", "pathname")`
- `pathname == "/settings"` のときだけ `theme = load_theme()` を取り、`theme-preview-store` と `theme-preview-name` を返す。
- それ以外は `PreventUpdate`（全体を一つの関数にするか、不要時は `PreventUpdate`）。

**`allow_duplicate`**: 既存の `update_preview_from_card` が同じ Output を触るため、両方 `allow_duplicate=True` が必要。

**`suppress_callback_exceptions`**: 既に [`app.py`](app.py) で有効。`/settings` 以外では `theme-preview-store` がレイアウトに無いが、既存の `register_theme_callbacks` も同じ Output を参照しているため、同じパターンで登録可能。

## Step 4: 防御的な小さな堅牢化（任意だが推奨）

[`mark_active_card`](components/theme_utils.py) 先頭で、`selected` が **dict**（誤って `theme-store` 形が入った場合）なら `selected.get("theme")` に正規化し、**常に文字列とカード id を比較**する。現状は文字列のみだが、将来のミスを防ぐ。

## Step 5: 動作確認（手動）

1. Supabase の `theme_settings` を **cerulean でも minty でもないテーマ**（ユーザー環境の実値）に固定し、可能ならローカル **`theme.txt` を意図的に cerulean 等にずらす**（経路 B と A のズレを再現しやすい）。
2. ブラウザで `/settings` を直接開く → **「選択中」・active カード・画面の Bootswatch** がすべて **DB の実テーマ**で一致し、**誤って cerulean 表示にならない**こと。
3. 別テーマをクリックしてプレビュー → 保存 → 再読み込み → 一貫していること。
4. プレビューのみで他ページへ → 戻る → 保存テーマに戻ること（既存の pathname 同期と整合）。

## Step 6: 自動検証

リポジトリルートで [post-change-verify](.cursor/skills/post-change-verify/SKILL.md) に従い `compileall` と `pytest tests/` を実行する。
