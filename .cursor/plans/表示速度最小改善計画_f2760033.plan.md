# 表示速度の最小改善計画（見た目維持）

## 目的

- デザイン（色付き主要カード）は維持したまま、体感速度の低下を抑える。
- 変更は最小限にし、影響範囲を `assets/styles.css` と `features/review` 周辺に限定する。

対象ファイル:
- [assets/styles.css](c:\Users\ryone\Desktop\oshi-app\assets\styles.css)
- [features/review/components.py](c:\Users\ryone\Desktop\oshi-app\features\review\components.py)
- [features/review/controller.py](c:\Users\ryone\Desktop\oshi-app\features\review\controller.py)
- （確認用）[terminals/1.txt](c:\Users\ryone\.cursor\projects\c-Users-ryone-Desktop-oshi-app\terminals\1.txt)

## Step 1: CSSの高コストトランジションを限定化

`transition: all` をやめ、必要プロパティのみへ置換する。

- 置換対象（現状3箇所）:
  - `.upload-area`
  - `.camera-button`
  - `.input-custom`
- 置換方針:
  - `transition: background-color, border-color, box-shadow, transform` に限定
  - duration は現状維持（0.2〜0.3s）

期待効果:
- レイアウト/サイズ等の不要な再計算を減らし、モバイルでの描画負荷を低下。

## Step 2: hover影の段差を1段弱める

`card-main-*` / `card-custom` / `photo-card` の hover 時 `--app-shadow-2` を弱い値へ変更する。

- 変更方針:
  - `--app-shadow-2` を1段弱く（例: `0 6px 16px rgba(...)`）
  - 既存の hover 実装はそのまま、トークン値だけ調整

期待効果:
- カードが多い画面でのホバー再描画コストを抑える（見た目の変化量も維持）。

## Step 3: レビューの Interval を「必要時のみ動作」へ

現状は [features/review/components.py](c:\Users\ryone\Desktop\oshi-app\features\review\components.py) の `io-intelligence-interval` が `disabled=False` で常時有効。

- 改修方針:
  - 初期は `disabled=True` に変更
  - `registration-store.tags.status == "loading"` の時だけ `disabled=False` に切り替えるコールバックを追加
  - `status in {idle, done, error, not_ready}` に戻ったら即 `disabled=True`

期待効果:
- `/_dash-update-component` の常時POSTを削減し、体感遅延を改善。

## Step 4: process_tags の再実行ガードを強化

[features/review/controller.py](c:\Users\ryone\Desktop\oshi-app\features\review\controller.py) の `process_tags` は `registration-store` 入力で呼ばれるため、同一状態での再入を抑制する。

- 改修方針:
  - `tags.status != "loading"` で早期 `PreventUpdate`（既存）に加え、
  - `front_photo.description_status == "processing"` 時の重複実行防止フラグを state に持たせる
  - `_update_tags` 実行後に必ず終端状態へ遷移（`done/error/not_ready`）し、loadingの取り残しを防ぐ

期待効果:
- 不要な連鎖 callback を止め、ログの連続POSTを減らす。

## Step 5: 画面別の軽量化確認（最小QA）

確認対象:
- `/register/barcode`
- `/register/photo`
- `/register/review`
- `/`（ホーム）
- `/gallery`

確認観点:
- 見た目（色付き主要カード）が維持されている
- フォーカス/再試行UIが変わらず機能する
- `terminals/1.txt` で idle時の `POST /_dash-update-component` 密度が下がる

## Step 6: 失敗時のロールバック方針

- 速度改善後に見た目が崩れた場合は、まず Step 2（shadow弱化）のみ戻して比較。
- 次に Step 1（transition限定）を段階的に戻し、どちらがボトルネックかを切り分ける。

## 完了条件

- `transition: all` が対象箇所から撤廃されている
- hover影が1段弱化され、主要カードの視認性は維持される
- `io-intelligence-interval` は loading 時のみ有効
- ログ上の不要な `/_dash-update-component` 連続POSTが減る
